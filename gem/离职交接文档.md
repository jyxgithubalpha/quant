## 一、项目概述

GEM 是一个端到端的量化机器学习实验框架，支持从数据读取、特征工程、模型训练、超参优化到回测评估的完整流程。框架采用 Hydra 配置驱动 + 依赖注入的设计，支持多种模型后端（GBDT / sklearn / PyTorch），并提供跨时间窗口的滚动状态管理。

核心价值：研究员只需修改 YAML 配置即可切换模型、调参策略、数据源和评估方式，无需改动代码。

> **注意**：本目录为 v1 版本。v2 版本位于 `/home/user165/workspace/quant/gemv2/`，架构有较大重构，详见 gemv2 交接文档。

## 三、核心数据流

```
YAML 配置 → Hydra 组合 → bootstrap.py → RuntimeBundle → ExperimentManager.run()
  │
  ├─ [1] SplitGenerator.generate()
  │       → SplitGeneratorOutput (splitspec_list, date_start, date_end)
  │
  ├─ [2] DataModule.prepare_global_store(date_start, date_end)
  │       → FeatherReader.read_sources()          # polars DataFrames
  │       → SingleSource 预处理 (DropNaN, FillNaN, DateFilter, CodeFilter, ...)
  │       → MultiSource 预处理 (AlignPreprocessor)
  │       → FeatureAssembler.assemble()            # → GlobalStore (numpy)
  │
  ├─ [3] 构建 SplitTask 列表
  │
  ├─ [4] 初始化 RollingState (空)
  │
  ├─ [5] DynamicTaskDAG → Executor → 每个窗口:
  │       SplitRunner.run():
  │         GlobalStore.take(indices) → SplitViews (train/val/test)
  │         MethodFactory.build()     → BaseMethod
  │         BaseMethod.run():
  │           a. TransformPipeline.fit_transform_views() → ProcessedViews
  │           b. UnifiedTuner.tune()   (n_trials > 0 时)
  │           c. Trainer.fit()         → FitResult
  │           d. Evaluator.evaluate()  → 指标
  │           e. ImportanceExtractor.extract()
  │           f. 构建 StateDelta
  │         → SplitResult
  │       更新 RollingState (EMA)
  │
  ├─ [6] ReportGenerator.generate()
  │
  └─ [7] run_visualization_pipeline() (若启用)
```

## 四、关键数据结构

| 类 | 位置 | 说明 |
|----|------|------|
| `GlobalStore` | `data/data_dataclasses.py` | 全局数据容器: X_full (np), y_full (np), keys (pl.DataFrame), `_date_to_indices` O(1) 日期索引 |
| `SplitView` | 同上 | 单分区 X/y/keys |
| `SplitViews` | 同上 | train/val/test 三元组 + split_spec |
| `ProcessedViews` | 同上 | Transform 后的视图 (含 feature_mask, transform_state) |
| `SplitSpec` | 同上 | split_id + train/val/test 日期列表 (int YYYYMMDD) |
| `TrainConfig` | `method/method_dataclasses.py` | 训练超参配置 |
| `FitResult` | 同上 | 训练结果: model, evals_result, best_iteration |
| `MethodOutput` | 同上 | 方法输出: 指标, 重要性, StateDelta |
| `StateDelta` | 同上 | 跨窗口传递: importance_vector, feature_names_hash, best_params |
| `RollingState` | `experiment/states/rolling.py` | 状态容器: Dict[state_key, BaseState], pickle 序列化 |

## 六、配置系统

根配置 `configs/config.yaml`:

```yaml
defaults:
  - splitgenerator: RollingWindowSplitGenerator
  - datamodule: defaults
  - method: lgbm
  - experiment: gbdt_cpu
  - _self_
seed: 42
```

### 常见修改场景

| 需求 | 操作 |
|------|------|
| 切换模型 | `method: lgbm → xgb / cat / ridge / mlp / ft_transformer` |
| 调整超参搜索范围 | `configs/method/tuner/<model>.yaml` |
| 修改数据源 | `configs/datamodule/sourcespec_dict/SourceSpec/` |
| 切换切分策略 | `configs/splitgenerator/` |
| 开启 GPU | `experiment: gbdt_gpu` |
| 启用调参 | `experiment/` 中 `n_trials > 0` |
| 启用可视化 | `experiment/` 中 `visualization.enabled: true` |
| 添加预处理 | `configs/datamodule/single_source_data_preprocessor_pipeline_dict/` |
| 修改 Transform | `configs/method/transform_pipeline/` |

### 数据源配置 (SourceSpec)

```yaml
_target_: gem2.data.data_dataclasses.SourceSpec   # 注意: 包名是 gem2
name: ret
format: feather
path: /path/to/data.fea
layout: tabular    # 或 "wide"
date_col: date
code_col: code
pivot: null         # "from_index" / "from_date" (wide 格式)
```

> **包名说明**: 目录名为 `gem/`, 但 Hydra `_target_` 引用 `gem2.*`。实际可导入包名为 `gem2`。

## 七、超参调优

```
UnifiedTuner = BaseSearchSpace + TunerBackend
```

| 后端 | 类 | 场景 |
|------|----|------|
| Optuna | `OptunaBackend` | 单机 (默认) |
| Ray Tune | `RayTuneBackend` | 分布式 |

关键特性:
- **Warm Start**: 从上一窗口最优参数初始化
- **Space Shrinking**: 缩小搜索范围到最优参数附近
- **并行 Trial**: `parallel_trials` 控制并发数

## 八、滚动状态 (RollingState)

跨时间窗口传递信息的核心机制:

| State | 功能 | 下游消费者 |
|-------|------|-----------|
| `FeatureImportanceState` | 特征重要性 EMA | FeatureWeightTransform |
| `TuningState` | 保存最优超参 | UnifiedTuner warm start |
| `SampleWeightState` | 样本权重 | 训练时样本加权 |

更新策略 (`StatePolicyConfig.mode`):
- `none` — 无状态, 每窗口独立
- `per_split` — 每窗口串行更新 (默认)
- `bucket` — 按季度分组, 组内并行, 组间串行

EMA 平滑系数: `ema_alpha` (默认 0.3)

## 十一、v1 设计特点与局限

### 设计特点
- Hydra `_target_` 驱动, 纯 YAML 切换组件
- 只支持 Feather/IPC 格式 (polars `read_ipc`)
- 日期为 int YYYYMMDD, 内部 `pl.Datetime("us")`
- 多源特征命名: `"source__column"` 避免冲突
- GlobalStore 全内存, 适合中等规模数据

### v1 局限 (v2 已解决)
- 数据结构定义分散在 `data/data_dataclasses.py` 和 `method/method_dataclasses.py`, 存在循环依赖风险
- Transform 管道嵌套在 `method/base/transforms/`, 耦合度高
- GBDT 三框架共享扁平文件 (`trainers.py`, `adapters.py`), 单文件过大
- sklearn 模型共享单一 Trainer, 缺乏每模型独立配置
- 无 Ensemble Pipeline, 不支持多模型组合
- `BaseMethod` 是继承式设计, 扩展需要子类化
